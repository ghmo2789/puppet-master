The Project

Learn offensive security to become better at defensive

The main goal of this project is to build a tool that allows you to control a network of computers, build an interface used to control the network and generally increase security awareness among developers. The tool is a server management system that consists of a web GUI with an associated web server, a control server software, and a client application, that together will allow a single user to control many computers from the web GUI. The system could, for example, be used for distributed computations, extracting or sending files to the controlled clients, or generating web traffic from multiple hosts to one website of choice. In other words, the tool that is going to be built during this project can be seen as a botnet manager that is intended to be used for educational and research purposes. The goal is to build an easy-to-use tool that will allow developers or system administrators to test their own systems and make them a bit safer against the cyber security threats that come from botnets.

Current Progress

With the first proper sprint completed, the project is starting to take form.

GUI

Below you can see part of our GUI. This will display the current connected clients and enable the user to select clients and a task for them to perform. We are still using dummy-data for this, and the current sprint will be used to connect these to the server.


Client

The initial plan for this week was to decide what framework to use for the client and implement a first version of the client that "calls home" to the control server by polling. The choice of framework landed on Rust, as this language allowed us to have the same code base for the application that will build and run on Linux, MacOS, and Windows. Regarding the implementation of the first version of the client we managed to implement the planned functionality, but we also managed to implement the functionality that was planned for the next sprint. This functionality is that the client can receive terminal tasks from the control server, run these commands and return the result to the control server. However, the current implementation do not allow the control server to send terminal tasks that is a blocking task, i.e. a task that do not end until user input is received.
